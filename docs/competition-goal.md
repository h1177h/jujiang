# 剧匠 Competition Goal

项目名：剧匠 / Jujiang
远程仓库：https://github.com/h1177h/jujiang

## 总目标

在当前 GitHub 仓库中完成一个可参赛、可演示、可解释开发过程，并且有真实产品竞争力的「AI 小说转剧本工具」。剧匠不应只是把题目要求逐条做完的 demo，而应该像一个小说作者真的愿意打开使用的创作工作台：输入小说后，能得到结构清晰、可追溯、可编辑、可校验、可继续打磨的剧本初稿，并能帮助作者理解改编过程中的人物、场景、冲突和节奏。

你需要自主规划技术栈、实现路线、PR 切分和提交节奏，不要机械执行固定流程。所有决策都要围绕比赛评分标准、剩余时间、可交付风险和产品完整度动态调整。

## Goal 模式使用说明

执行者不能只按字面 checklist 完成任务，而要持续做产品判断：如果当前实现看起来简陋、像硬编码 demo、缺少真实 API、缺少非 YAML 编辑，或只是把题目要求逐项摆出来，就应主动把它视为未完成，并优先推进结构性改造。

建议启动语：

`/goal 严格读取并遵循 D:\jujiang\docs\competition-goal.md，自主规划并完成剧匠产品级优化。不要直推 main，按有意义的阶段创建英文 Conventional Commits 标题、中文真人风格描述的 PR。`

## 产品定位

剧匠的核心用户是希望把小说改编成短剧、微短剧、分场剧本或影视脚本初稿的小说作者、编剧助理和内容团队。产品重点不是生成视频，而是把小说文本转化为可编辑的剧本结构，并让作者能快速判断“这版剧本是否能继续打磨”。

产品应体现三个关键词：

- 可控：作者可以选择改编风格、查看结构、编辑 YAML，而不是只得到一段黑盒输出。
- 可追溯：剧本场景应尽量能回到原章节或原文片段，方便作者核对改编是否偏离原作。
- 可打磨：输出不是一次性结果，而是一个可校验、可修改、可导出的剧本工作底稿。

## 核心体验

最终体验应尽量接近一个完整创作工作台，而不是单页表单。理想使用路径：

1. 作者导入或粘贴小说正文；可以是一段短篇素材，也可以是 3 章以上长文。比赛 demo 必须覆盖 3 章以上，但产品体验不能把它做成普通用户的硬限制。
2. 系统识别章节、提取主要人物、地点、情绪线索和关键事件。
3. 作者选择改编目标，例如短剧、影视场景、分场大纲、对白强化或悬念强化。
4. 系统生成结构化剧本 YAML，并同时提供人类可读的场景卡片或剧本预览。
5. 作者能查看每个场景对应的章节来源、冲突强度、人物关系或节奏提示。
6. 作者能编辑 YAML，系统实时或手动校验结构。
7. 作者能复制、下载或导出剧本初稿，用于后续打磨。

如果时间不足，优先保证这条路径的完整性和可信度，而不是堆砌松散功能。

## 竞争力方向

在满足题目硬性要求之外，剧匠需要至少形成 2-3 个有记忆点的产品亮点。你可以根据实现难度自主选择，但要优先选择能在 demo 中清晰展示、能服务小说转剧本本身的能力。

可选方向：

- 改编控制台：支持选择剧本类型、叙事节奏、对白密度、悬念强度或单场时长倾向。
- 场景卡片预览：把 YAML 场景渲染为可读卡片，展示地点、时间、人物、目标、冲突、对白摘要。
- 原文追溯：每个场景带章节来源、原文摘要或引用定位，让作者知道改编依据。
- 人物关系与弧光：自动生成角色表、人物关系、角色目标和变化轨迹。
- 冲突与节奏分析：给场景标注冲突强度、情绪走向、节奏提示，帮助判断剧本是否好看。
- YAML 质量仪表盘：显示章节覆盖率、场景数量、对白占比、缺失字段、校验状态。
- Fallback 创作引擎：没有 API key 时也能生成结构合理的示例剧本，保证评委一定能跑通。

不要为了“创新”去做偏离主题的大功能，例如完整视频生成、复杂素材库、重型 Electron 桌面端或多模型平台。竞争力应来自小说到剧本工作流的完成度、可控性、可追溯性和表达清晰度。

## 产品级重做要求

当前目标不是做一个“满足题目 checklist 的 demo”，而是做一个有竞争力的小说改编工作台。开发时必须主动挑战当前实现是否简陋，不能因为某个功能已经“存在”就认为完成。

如果现有实现方向太弱，可以重构、替换甚至删除重写。保留旧代码不是目标，做出可信产品才是目标。

### 必须避免的低质形态

- 不要把产品做成左右两个 textarea：左边小说，右边 YAML。
- 不要只靠“场景卡片”冒充完整产品。场景卡片只是展示层，不是完整创作流程。
- 不要把本地草稿规则引擎包装成“AI”。本地草稿只能是无 API key、网络失败或快速起稿时的兜底能力。
- 不要在产品 UI 里写“比赛要求”“提交标准”“至少 3 章才能生成”这类面向评审的提示。评审约束只能出现在 demo 文档或提交材料里，不能污染作者使用体验。
- 不要机械一章拆一场或一章固定拆两场。分场应基于段落、事件转折、地点/时间变化、人物行动、对白和冲突。
- 不要让作者只能编辑 YAML。YAML 是高级交付格式，不应是唯一创作界面。
- 不要写模板化 PR 描述或最终说明。文字要像真人开发记录，具体、简洁、说明真实取舍。
- 不要为了凑创新点放静态说明卡。创新点必须能在 demo 中操作或被验证。

### 产品必须像工作台

最终产品应至少包含这些工作台能力：

1. 原文导入区：支持粘贴、上传、示例小说、章节/段落识别状态；短文本也能生成草稿，质量不足只作为建议提示，不阻断。
2. 改编设置区：支持改编类型、风格、节奏、对白密度或悬念强度等可控参数。
3. AI 生成区：支持真实 OpenAI-compatible API；显示 provider、model、生成状态、失败原因和本地草稿兜底原因。
4. 故事分析区：展示章节摘要、关键事件、角色关系、冲突/节奏曲线、来源覆盖率。
5. 场景工作区：不是只读卡片，而是可选择、可编辑、可审稿的场景列表。
6. 场景编辑器：能编辑场景目标、地点、时间、人物、动作、对白、转场、冲突等级、修订建议，并能同步到 YAML。
7. 原文追溯区：选中场景后能看到对应原文段落、行号、摘录或高亮。
8. YAML 交付区：保留可编辑 YAML、Schema 校验、复制、下载。
9. 质量检查区：指出缺失字段、低冲突场景、无对白场景、来源覆盖不足、疑似角色/对白归属错误。

### API 是一等能力

真实 AI API 不是可选装饰，而是核心能力。要求：

- 支持 OpenAI-compatible `/v1/chat/completions` 或更合适的兼容接口。
- 用户可以配置 Base URL、API Key、Model。
- API 生成必须优先于本地草稿。
- API 返回必须经过结构校验；校验失败要提示原因，不要静默吞掉。
- 本地草稿只在未配置 key、网络失败、API 失败、快速起稿或现场无网时使用。
- UI 必须明确显示当前结果来自 AI 还是本地草稿，但文案要面向作者，不要像工程日志或比赛说明。
- 如果前端直连 API 存在 CORS 或 key 暴露风险，应在文档中说明；时间允许时优先实现轻量后端 proxy。

### 非 YAML 编辑是硬要求

YAML 是交付格式，不是产品体验本身。必须提供场景级编辑体验：

- 作者可以不看 YAML，也能修改剧本内容。
- 场景编辑后，要能同步更新 YAML，或提供明确的“应用到 YAML”动作。
- YAML 校验失败时，应该能定位到对应场景或字段。
- 不能只做“YAML textarea + 预览”，这不算完整产品。

### 真正有竞争力的创新点

至少实现 3 个可操作、可验证的亮点，优先从以下选择：

- 角色关系图或关系矩阵。
- 章节到场景的可视化映射。
- 冲突/节奏曲线。
- 原文追溯高亮。
- 场景质量检查清单。
- AI 结果与本地草稿结果对比。
- 同一段文本在不同改编风格下的差异预览。
- 场景版本历史、撤回或局部重生成。
- 一键补强某个场景：补对白、强化冲突、压缩节奏、改写转场。

这些亮点必须服务小说转剧本本身。不能做和题目无关的装饰功能。

## 比赛题目

开发 AI 小说转剧本工具，能将 3 个章节以上的小说文本自动转换为结构化剧本 YAML，使作者获得可编辑、可继续打磨的剧本初稿。必须额外写一篇文档定义剧本 YAML Schema，并说明该 Schema 的设计原因。

## GitHub 与 PR 规范

开始前先检查：

- 当前目录是否为 `h1177h/jujiang` 仓库。
- `git remote -v` 是否指向 `https://github.com/h1177h/jujiang` 或等价 SSH 地址。
- 当前分支和工作区状态。
- `gh auth status` 是否可创建 PR。
- 如果当前目录不是目标仓库，先安全 clone 或提示我确认，不要覆盖已有文件。
- 不要直接推 `main` / `master`。

你需要边开发边创建 PR。PR 是协作、审查和展示开发过程的载体，不是单纯的提交计数。不要固定 PR 数量，也不要机械套模板；根据当前阶段自主判断 PR 的粒度。倾向创建小而聚焦、上下文清楚、可以独立审查和验证的 PR。一个 PR 可以完成一个功能、一个架构层、一次质量改进、一次文档沉淀或一次演示增强；如果一个 PR 很难用一句英文标题说清楚，通常应该考虑拆分。如果改动还在探索中，可以先使用 draft PR 或保留本地分支，等能说明价值和验证方式后再正式创建。

PR 标题必须使用英文，优先采用 Conventional Commits 风格：

`<type>(optional-scope): <short summary>`

标题要能让评审快速理解这个 PR 改了什么。`type` 和 `scope` 由你根据实际改动选择，不要为了套格式牺牲准确性。常用类型包括 `feat`、`fix`、`docs`、`test`、`refactor`、`chore`、`ci`、`style`。scope 使用简短英文小写词，例如 `ui`、`parser`、`schema`、`converter`、`docs`、`demo`、`test`。

标题示例：

- `feat(ui): build screenplay workspace`
- `feat(parser): add chapter segmentation`
- `docs(reference): summarize adaptation strategy`
- `test(schema): cover screenplay validation`

注意：PR 标题使用英文冒号 `:`，不要使用中文冒号。标题应简洁、具体，避免 `update code`、`fix bug`、`final changes` 这类无法表达价值的标题。

PR 描述必须使用中文，但不要求每次都机械包含同一套栏目。描述的目标是让评审不用猜：为什么做、做了什么、怎么验证、还有什么风险。根据 PR 类型选择合适的信息，保持简洁但完整。

建议覆盖的信息：

- 变更目的：这个 PR 解决了什么问题，推进了哪项评分标准。
- 主要改动：列出核心文件、模块或用户可见变化。
- 验证方式：写清楚运行过的命令、结果；如果只能手动验证，说明手动步骤。
- 参考与改写：如果本 PR 借鉴了参考项目，说明借鉴点和剧匠中的独立改写方式。
- 截图或演示说明：UI、交互、导出、demo 相关 PR 需要提供。
- 风险与后续：说明未覆盖的边界、时间取舍或下一步。

对于纯文档、纯测试、很小的修复 PR，可以使用更短描述；对于架构、核心功能、UI 或演示相关 PR，描述必须更完整。无论长短，都必须包含验证结果或说明无法验证的原因。

如果环境无法创建 PR，则完成分支和 commits，并输出可复制的 `gh pr create` 命令和完整中文 PR 正文。

## 必须完成的产品能力

1. 支持输入或上传小说文本；短篇片段可生成草稿，比赛演示样例必须覆盖至少 3 个章节。
2. 自动识别章节并清洗文本。
3. 支持真实 API 生成结构化剧本；本地草稿只能作为无 key、快速起稿或失败兜底。
4. YAML 至少包含：作品元信息、角色表、章节映射、场景列表、场景目标、地点、时间、出场人物、动作描写、对白、旁白/转场、情绪/冲突、原文来源定位。
5. 提供非 YAML 的场景级编辑界面，能编辑场景目标、地点、时间、人物、动作、对白、转场、冲突等级和修订建议。
6. 场景编辑能同步到 YAML，或有明确的应用/保存动作。
7. 提供可编辑 YAML 区域，支持复制和下载 YAML。
8. 提供 YAML Schema 或结构校验，并能提示错误位置和原因。
9. 提供示例小说输入和示例输出，确保无 API key 时也能 demo。
10. 写 `docs/yaml-schema.md`，定义字段、必填/可选、示例和设计原因。
11. 写 `README.md`，包含安装、运行、API 配置、功能、架构、演示步骤。
12. 至少加入 3 个有展示价值且可操作的创新点，例如角色关系图、章节到场景映射、冲突/节奏曲线、原文追溯高亮、场景质量检查、AI/本地草稿对比、局部重生成。
13. 添加必要测试，至少覆盖章节解析、短文本/无标题文本生成、空输入错误态、场景拆分、API provider、YAML 生成结构、Schema 校验、场景编辑同步、本地草稿兜底路径。

## 产品质量门槛

不要满足于“功能存在”。每个核心能力都要尽量达到可演示、可理解、可继续扩展的水平：

- 输入体验：示例文本、空状态、错误提示、章节不足提示要清楚。
- 转换体验：生成过程要有状态反馈，明确显示 AI 或本地草稿，失败时给出原因和本地草稿兜底状态。
- YAML 体验：结构清晰、字段命名稳定、缩进正确、可读性好。
- 校验体验：不要只显示“失败”，要指出缺失字段或错误位置。
- 编辑体验：评委不应该必须读 YAML 才能理解和修改作品，应提供可操作的场景工作区。
- 预览体验：场景卡片、摘要或结构化预览只能辅助理解，不能替代编辑和审稿流程。
- 文档体验：README、Schema 文档、参考分析和 demo 脚本应能支撑评审理解产品价值。
- 工程体验：核心逻辑应模块化，解析、转换、Schema、校验、UI 展示、导出尽量分离。

## 技术策略

如果仓库为空或技术栈不明确，优先选择 1.5 天内最稳的轻量方案，例如 Vite + React + TypeScript + Zod + YAML。真实 AI API 是一等能力，本地草稿/mock 转换只用于保证 demo 可跑和离线起稿。

不要被参考项目带偏去做完整视频生成、Electron 大工程、素材管理或复杂后端，除非仓库已有基础且时间允许。优先保证小说到剧本 YAML 的完整闭环、Schema 文档、可编辑体验、校验体验和 demo 表达。

如果前端直连 API 影响真实可用性，应评估并实现轻量 API proxy。proxy 必须保持简单，不要引入重型后端，除非产品需要。

## 自主规划要求

你需要自己判断每一步最值得做什么。每次开始新阶段前，先基于当前状态回答：

- 当前最影响评分的缺口是什么？
- 当前最影响产品竞争力的缺口是什么？
- 当前实现有哪些地方看起来像 demo、硬编码或 checklist？
- 这个阶段应该做成什么可验证结果？
- 是否值得单独开 PR？
- 这个 PR 如何体现产品完整度、代码质量或演示价值？

不要等待我给固定任务列表。遇到时间不足时，优先保证可演示闭环、真实 API 能力、非 YAML 编辑体验和评审材料。

## PR 前自检循环

每次创建 PR 前必须自检：

1. 这个 PR 是否有独立价值？
2. diff 是否清晰、范围是否合理？
3. 是否能运行、构建或测试验证？
4. 是否包含无关文件、缓存、密钥、临时文件？
5. PR 标题是否英文且格式规范？
6. PR 描述是否中文且包含测试结果？
7. 是否存在照搬参考项目代码、Prompt、Schema、文案或样式的风险？
8. 是否推进了评审标准中的至少一项？
9. PR 描述是否像真人开发记录，而不是 AI 模板？
10. 这个 PR 是否真的让产品更完整，而不是只补 checklist？

如果自检不通过，先修复再 PR。

## 最终交付自检

最终交付前必须完成：

- 本地能完成完整 demo：输入 3 章小说 -> 配置或跳过 API -> 生成剧本 -> 场景级编辑 -> 同步/查看 YAML -> 校验通过 -> 复制/下载 -> 查看 YAML Schema 文档 -> 展示创新点。
- API 生成可用或有清晰的环境限制说明；本地草稿只作为兜底和离线起稿。
- 非 YAML 场景编辑工作流可用。
- 至少 3 个创新点可操作、可验证。
- README、`docs/yaml-schema.md`、`docs/reference-analysis.md` 完整。
- 测试/构建命令已运行并记录结果。
- 所有 PR 链接、commit 分布、运行命令、测试结果、架构说明、demo 视频脚本、剩余风险已汇总。
- 输出一份适合比赛提交的最终说明，重点对应 40% / 40% / 20% 评分标准。

最终回答中还要用产品视角说明：

- 剧匠解决了小说作者改编剧本时的什么痛点。
- 相比普通 AI 对话生成剧本，剧匠的工作台优势是什么。
- 哪些能力体现了完整度，哪些能力体现了创新性。
- 架构如何支撑后续扩展真实 AI、多轮改写、更多导出格式或团队协作。

## 执行原则

持续根据评审标准判断优先级，但不要只做“题目合规”的最低实现。遇到取舍时，优先选择能让产品显得完整、可信、有作者使用价值的方案。遇到阻塞不要空转，说明当前状态、失败原因、替代方案，并继续推进最小可行路径。能用简单可靠方案完成闭环时，不要引入重型架构。能通过文档和 demo 清楚表达的亮点，要及时沉淀到 README、docs 和 PR 描述里。

如果用户批评“简陋”“像 demo”“只是按要求做”，必须把它视为产品标准没有达成，而不是解释已有功能。下一步应先做产品审计，再做结构性改造。
