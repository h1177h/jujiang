# 参考项目分析

本文记录剧匠对参考项目的调研结论。调研只吸收产品流程、模块边界、数据建模和演示表达的思路；没有复制、近似复制或 vendor 任何参考项目源码、Prompt、Schema、README、样式、组件实现或目录结构。

## 调研边界

比赛题目是“AI 小说转剧本工具”，不是完整视频生成平台。因此剧匠只取与小说到结构化剧本 YAML 有关的轻量设计：

- 文本输入后的章节识别与清洗。
- 角色、场景、对白、冲突、来源定位等结构化字段。
- 可编辑工作台和校验反馈。
- 无 API key 时仍可查看和编辑示例结果，但不伪造剧情生成。

不吸收的部分：

- 完整视频生成链路。
- Electron 大工程。
- 复杂素材库和后台管理系统。
- 第三方项目的 Prompt、Schema、组件和目录结构。

## openfrees/toonflow

项目链接：https://github.com/openfrees/toonflow

值得借鉴的点：

- 产品定位直接覆盖“AI 写剧本”和“小说转剧本”。
- 提供前后端、桌面端和 Docker 部署，说明成熟工具通常会把创作流程包装成完整工作台。
- README 中强调“小说项目、章节、剧情线到剧本”的转换流程，这对剧匠的章节映射有启发。

转化进剧匠的设计：

- 剧匠保留“章节 -> 场景”的核心链路，但不引入后台、桌面端或 Docker。
- 将章节映射沉淀为 `chapterMappings`，让 YAML 明确说明每章生成了哪些场景。

独立改写：

- 剧匠不采用该项目的服务端、Electron 或目录结构；自动生成必须依赖真实 AI provider。
- 剧匠输出目标是可编辑 YAML，而不是完整短剧生产系统。

## HBAI-Ltd/Toonflow-app

项目链接：https://github.com/HBAI-Ltd/Toonflow-app

值得借鉴的点：

- 围绕“策划 -> 编剧 -> 分镜 -> 出片”组织工作流。
- 无限画布、Agent 协作、章节事件图谱和可编程供应商系统展示了成熟产品如何处理复杂创作任务。
- 代码树里能看到 `src/routes/task/*`、`src/socket/routes/scriptAgent.ts`、`src/routes/novel/event/*` 这类任务和 agent 通道，说明长流程不会只靠按钮后的单句提示。
- 章节事件图谱说明长文本改编需要先结构化事件，再调用上下文生成剧本。

转化进剧匠的设计：

- 剧匠没有做无限画布，而是把“事件图谱”的思想缩小为 `chapterMappings`、`scenes.source` 和 `rhythmStats`。
- 本轮进一步把“章节事件图谱”落成 `chapterEvents`，并在 AI 调用中先生成故事蓝图，再生成最终剧本 YAML。
- 长篇生成不再只做一次大 prompt；超过 3 章时会逐章抽取事件，再合并故事圣经和改编策略。
- 可编程供应商系统被转化为更轻的方向：保留 `generatedBy` 字段，为后续 AI provider 插拔预留位置。
- Agent 协作和局部任务的思路被转化为“单场 AI 补强”，只重写当前 scene，不重跑整篇剧本。
- 任务和 socket 状态的思路被转化为更轻的生成任务面板：只记录剧匠自己的连接、事件抽取、故事圣经、剧本生成、Schema 修复和 YAML 写入状态。

独立改写：

- 剧匠不复用 Toonflow-app 的 Agent、记忆、供应商或 Skill 实现。
- 剧匠的 UI 是小说改编工作台，服务于 YAML 生成、审稿、校验和继续改写，不追求完整视频生产。

## Anning01/novelvids

项目链接：https://github.com/Anning01/novelvids

许可证状态：GitHub 仓库元信息未显示明确 licenseInfo，因此剧匠只做概念层参考，不引入代码或文案。

值得借鉴的点：

- 以章节为单位组织四步 AI 工作流：实体提取、资产管理、分镜生成、视频合成。
- 小说管理中包含元数据、章节拆分和工作流状态追踪。
- 工程结构强调分层和测试，对比赛中的代码质量维度有参考价值。

转化进剧匠的设计：

- 剧匠采用章节级处理单位，但只保留剧本生成前半段：章节解析、角色提取、场景生成。
- 把“实体提取”转成 `characters` 和 scene 的 `characters` 字段。
- 把“工作流状态”简化成实时 Schema 校验结果。

独立改写：

- 剧匠没有引入 FastAPI、数据库、模型配置或视频库。
- 剧匠不使用 novelvids 的 prompts、schemas 或服务层代码。

## alibaba/lumenx

项目链接：https://github.com/alibaba/lumenx

值得借鉴的点：

- LumenX 将小说文本到动态视频拆为资产提取、风格定调、资产生成、分镜脚本构造、分镜图生成和视频生成。
- 它强调深度剧本分析：自动提取角色、场景、道具，并生成可编辑分镜脚本。
- 演示流程清楚地分成 Script、Art Direction、Assets、StoryBoard、Motion 等步骤。
- 最新调研中看到 `GenerationBanner`、`TaskQueuePanel`、`PendingTaskAffordance` 等 UI 组件，说明成熟创作工具会把长任务的阶段、队列、失败和重试入口显式暴露给用户。

转化进剧匠的设计：

- 剧匠吸收“先结构化，再生成”的流程：先抽取章节事件、故事圣经和改编策略，再生成剧本 YAML，不进入图像或视频阶段。
- 将“风格定调”缩小为 `adaptationStyle`：均衡、影视感、舞台、短剧。
- 将“可编辑分镜脚本”转成可编辑 YAML 区域和实时校验。
- 将“任务队列/生成横幅”的概念收敛为轻量生成任务面板：剧匠不做跨项目队列，但让作者知道当前 AI 调用走到哪一步。
- 将“失败任务可回看、可重试”的交互缩小为最近生成记录：刷新页面后仍能看到上一次失败卡在哪个阶段，并用当前 AI 配置重新生成。
- 将“创作过程可回看”的思路缩小为版本对比：不做复杂协同审稿，只展示当前 YAML 相比旧版本的新增、删除和变更行。

独立改写：

- 剧匠没有接入 Qwen、Wanx、FastAPI、Three.js 或 FFmpeg。
- 剧匠的 Schema 与 LumenX 无关，只服务小说转剧本 YAML。

## xuanyustudio/LocalMiniDrama

项目链接：https://github.com/xuanyustudio/LocalMiniDrama

值得借鉴的点：

- 强调本地运行、数据不出本机、开箱即用。
- 完整流程包含故事生成、剧本编辑、角色生成、场景生成、分镜生成、图片/视频生成和合成。
- 一键流水线提供进度展示、错误日志、失败重试和补全缺失内容。
- `taskService.js`、`generationTaskStore.js` 和 `useGenerationTaskSync.js` 展示了本地产品如何把异步任务状态持久化、同步和清理。

转化进剧匠的设计：

- 剧匠优先保证无 API key 时仍可 demo 示例 YAML 的编辑和校验，但不在本地生成剧情剧本。
- 将“进度和错误反馈”的思想转化为 AI 阶段状态、YAML 校验提示和 Schema 修复回合。
- 将“异步任务状态”的思想进一步转化为前端 `generationRun`：记录本次生成的阶段、耗时、失败原因和完成状态。
- 将“失败重试和补全缺失内容”的思想缩小到单场补强和版本快照，方便作者在局部继续打磨。
- 将“本机工作状态可恢复”的思路缩小为浏览器端工作区草稿：保存标题、原文、改编风格、YAML、选中场景和版本历史，避免刷新页面后丢失创作现场。
- 将“异步任务状态持久化”的思路进一步缩小为生成记录：最近的成功和失败任务会随工作区草稿保存，失败任务保留阶段、错误和重试入口。
- 将“失败后继续打磨”的思路延伸到版本历史：作者可以先比较 AI 补强或手动修改前后的 YAML 差异，再决定恢复哪一版。
- 将“剧本编辑”聚焦到可编辑 YAML，而不是完整分集/素材管理。

独立改写：

- 剧匠不做视频模型接入、资源库、工程导入导出或一键视频流水线。
- 剧匠前端和转换引擎均为比赛题目独立实现。

## no copied code statement

剧匠没有复制或近似复制上述项目的大段源码，没有搬运它们的 Prompt、Schema、README 文案、样式、组件实现或目录结构，也没有 vendor 任何参考项目代码。

其中 `Anning01/novelvids` 的许可证信息不明确，剧匠对它只做流程概念参考；其余项目即使许可证允许开源使用，剧匠也没有引入源码。

参考项目对剧匠的影响仅限于：

- 明确成熟产品通常采用“文本 -> 实体/事件 -> 场景/分镜 -> 生成物”的分层流程。
- 认识到长文本改编必须保留章节和来源追溯。
- 认识到长文本生成需要中间层，不能把整篇小说直接当成一次性 prompt。
- 认识到成熟流水线需要失败修复回合，不能让模型输出少字段时直接把用户丢进失败态。
- 认识到作者真正需要的是可局部修改和可回退的工作台，而不是每次修改都整篇重跑。
- 认识到真实 provider 调用需要可配置、可测试、可诊断的连接层，而不是把浏览器直连失败暴露成一句笼统的 `Failed to fetch`。
- 进一步对照 LocalMiniDrama、novelvids、Toonflow-app 和 LumenX 后，剧匠把 API 设置从“transport URL + key”调整为“Provider profile + 应用内 AI 服务”：作者只配置上游 Provider，页面固定调用本机应用服务，由服务端按每次请求携带的 Base URL 和 Key 转发。
- 认识到 demo 需要清楚表达输入、转换、编辑、校验和导出的闭环。

剧匠的独立选择是：把题目收敛到小说转结构化剧本 YAML，用轻量前端完成编辑、校验和交付闭环，自动生成交给真实 AI provider，而不是本地规则伪造剧情理解。API 调用层采用应用内 AI 服务作为固定路径，页面可记住 Provider、Base URL、Model 和 Key，服务端仍可读取环境变量作为演示环境默认值；生成前先做连接测试，把应用服务未启动、未读到 key、上游地址错误这几类问题分开提示。
