# 剧匠 Demo 脚本

目标：用 3 到 4 分钟展示“输入小说 -> 生成 YAML -> 校验 -> 编辑 -> 复制/下载 -> 解释创新点”的完整闭环。演示建议使用内置多章示例，也可以现场粘贴一段短篇素材证明产品不是只服务固定格式。

## 准备

```bash
npm install
npm run dev
```

浏览器打开 Vite 本地地址。建议录制前先运行：

```bash
npm audit
npm test
npm run build
```

## 讲解顺序

### 1. 开场

“这是剧匠，一个小说转结构化剧本 YAML 的轻量工作台。它的重点不是直接生成视频，而是先给作者一个可编辑、可校验、可追溯的剧本初稿。”

展示页面左侧示例小说《雾港来信》。

### 2. 输入与风格

说明左侧可以粘贴或上传 `.txt/.md` 小说文本；配置 AI 后，短篇片段和多章长文都可以生成剧本草稿。

切换一次改编风格，例如从“影视感”切到“短剧”，说明风格会影响场景目标和转场表达。

如果现场有 API key，优先展示本地 proxy 模式：

```bash
npm run proxy
```

然后在页面勾选“AI 生成”和“本地 proxy”，填写 API Key，按需勾选“记住 API Key”，再点击“测试连接”。默认 proxy 地址是 `http://127.0.0.1:18787/v1`；如果端口被占用，可以用 `JUJIANG_PROXY_PORT` 改端口，并同步页面 Base URL。如果不想在页面填 key，也可以先设置 `$env:JUJIANG_API_KEY="..."` 再启动 proxy。前端直连模式保留给临时测试，但很多 provider 会因为 CORS 或网络策略拒绝浏览器直连。

如果没有 key，只展示示例 YAML 的查看、编辑和校验，不演示自动生成。

顺手说明工作区会自动保存：标题、原文、改编风格、YAML、选中场景、版本历史和最近生成记录都会留在本机浏览器里，录制过程中刷新页面不会丢稿，也能回看上一次生成卡在哪个阶段。

### 3. 生成 YAML

点击“生成结构化剧本 YAML”。

生成过程中停留在左侧任务面板，说明它会记录本次 AI 调用的阶段：连接检查、事件抽取、故事圣经合并、剧本生成、Schema 修复和 YAML 写入。长篇小说不会只给一个转圈状态，失败时也能看出卡在连接、结构修复还是最终写入，并且可以用当前 AI 配置直接重试。

讲解右侧 YAML 的核心结构：

- `work`：作品元信息和改编风格。
- `adaptationPlan`：改编方向、语气和下一轮修订重点。
- `characters`：角色表和角色关系摘要。
- `chapterEvents`：AI 第一阶段抽取出的章节事件图谱。
- `storyBible`：故事圣经，包含核心冲突、时间线和角色弧光。
- `adaptationStrategy`：进入剧本生成前的分场和风险控制策略。
- `chapterMappings`：小说结构单元到场景的映射。
- `scenes`：每场戏的目标、地点、时间、人物、动作、对白、转场、情绪、冲突和来源定位。
- `rhythmStats`：场景数、对白数、平均冲突和高冲突场景。
- `storyDiagnostics`：来源覆盖、节奏概览和潜在风险。

补充说明长篇策略：短篇和三章示例会快速走事件图谱到剧本生成；更长的小说会先逐章抽取事件，再合并故事圣经和改编策略，避免把整本小说一次性塞进 prompt。若模型第一次返回的剧本结构不完整，剧匠会把 Schema 错误交给模型修复一次，再决定是否失败。

然后向下滚动到作者审稿台，展示同一份 YAML 被解析成改编计划、章节事件图谱、故事诊断、角色关系、节奏指标、章节到场景映射、冲突曲线、质量检查和分场卡片。

点击章节事件、scene chip、冲突柱或质量检查项，说明它们不是静态说明，而是能定位到对应场景的审稿入口。

### 4. 校验与编辑

先选中一个场景，在右侧场景编辑器里修改：

- 场景目标
- 地点或时间
- 对白
- 冲突等级
- 修订建议

展示右侧 YAML 已同步变化，节奏统计、冲突曲线和高冲突场景也会跟着更新。

如果现场有真实 API，点击场景编辑器里的“AI 补强”，说明它只把当前场景、前后场摘要、章节事件和故事圣经发给模型，不会整篇重跑。补强完成后展示 YAML 更新，并在“版本历史”里出现一个新快照。

在版本历史里点选旧版本，展示“与当前 YAML 对比”：新增、删除、变更行会分开展示。这里的价值是让作者知道 AI 补强或手动改稿到底动了哪些字段，而不是只能盲目恢复整份 YAML。

然后在 YAML 中临时删除一个必填字段，例如删除某个 scene 的 `goal`，展示校验失败提示。撤回或补回字段，展示“Schema 校验通过”。

### 5. 原文追溯

定位到任意 scene 的 `source`：

- `chapterIndex`
- `chapterTitle`
- `paragraphIndexes`
- `lineStart`
- `lineEnd`
- `excerpt`

讲解：“这让作者知道每场戏来自原文哪里，不是凭空生成。”

### 6. 复制与下载

点击保存版本、恢复一个历史版本，然后刷新页面，说明当前工作区和最近生成记录会自动恢复；再点重置工作区，说明它只清理草稿，不会动 API 设置。最后点击复制和下载，说明 YAML 可以交给作者继续编辑，也可以进入后续剧本或分镜流程。

### 7. 文档与质量

打开 README 和 `docs/yaml-schema.md`，说明字段设计和运行方式。

打开 `docs/reference-analysis.md`，说明参考项目只用于产品流程分析，没有复制代码、Prompt、Schema 或样式。

最后展示命令验证结果：

- `npm audit`：0 vulnerabilities。
- `npm test`：9 个测试文件、46 个测试用例通过。
- `npm run build`：构建通过。

## 备用说明

如果网络或 API key 不可用，只演示示例 YAML 的编辑、校验、复制和下载。自动生成必须依赖 AI，剧匠不使用本地规则伪造剧情理解。若页面提示 `浏览器直连失败`，切回本地 proxy；若提示 proxy 未连接，先启动 `npm run proxy` 并点击“测试连接”确认。
